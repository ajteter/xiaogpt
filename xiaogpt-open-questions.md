## 十、根据当前仓库代码的回答

说明：

- 以下结论只基于当前仓库里的源码、`README.md`、`xiao_config.yaml.example`
- 能从代码直接确认的，我会写成明确结论
- 不能从代码直接确认的，我会明确标注“代码无法证明”

### 1. `hardware`

- 代码无法推导出你家设备真实的 `hardware`，这必须靠设备底部标签、`micli mina` 或 `micli list` 获取。
- `hardware` 在代码里主要用于三件事：匹配设备列表、请求小爱最近对话接口、给 `use_command` 选择命令号。
- 如果 `hardware` 配错，启动时会在设备枚举阶段直接报错。

### 2. `mi_did`

- 代码同样无法直接给出你设备真实的 `mi_did`，部署前仍然需要外部获取。
- 如果没有显式配置 `mi_did`，且不是 `cookie` 路径，程序会尝试按 `model.endswith(hardware.lower())` 自动推断；推断失败就会报错。
- 如果你使用 `cookie`，源码会跳过 `_init_data_hardware()`，这意味着自动补全 `mi_did` 的逻辑也会一起跳过。
- 因此，只要你要用 `use_command`，或者要用“开始持续对话”里的唤醒命令，最好还是明确配置 `mi_did`。

### 3. 登录方式：`account/password` 还是 `cookie`

- 单看源码，`cookie` 不是完整的“免登录模式”。
- `init_all_data()` 总是先执行 `login_miboy()`，而 `login_miboy()` 总是调用 `MiAccount.login("micoapi")`。
- `cookie` 在当前代码里主要只影响两件事：跳过设备信息初始化、以及在 `get_cookie()` 里直接构造请求小爱接口用的 cookie。
- 这意味着：如果你打算在一个全新的 Docker 容器里只放 `cookie`，既不提供账号密码，也不持久化 `~/.mi.token`，当前代码并不能证明这种方式一定能稳定启动。
- “账号密码更稳定”还是“cookie 更稳定”、是否会遇到风控，这类结论代码无法证明。

### 4. `change_prompt_keyword` 的实际行为

- 结论一：修改只在当前运行进程内生效，不会写回 `config.yaml`，重启后丢失。
- 结论二：不在持续对话模式时，说“更改提示词 ...”通常只会改 prompt，不会顺手再问模型。
- 结论三：在持续对话模式里，这句命令会先改 prompt，再继续走当前轮处理流程，所以新 prompt 可能立即影响当前轮。
- 结论四：这条命令在持续对话里还有一个边界行为：控制命令文本本身也可能被当作当前问题继续发给模型。
- 代码里的真实修改动作是：去掉触发词后，在前面强行加上“以下都”，再更新内存中的 `self.config.prompt`，然后调用 bot 的 `change_prompt()`。

### 5. `use_command` 什么时候必须开启

- 源码里没有“按型号自动强制开启”的逻辑，只有一个布尔开关。
- 开启后，TTS 会改走 `miio_command`；不开启则走 `mina_service.text_to_speech()`。
- 仓库 README 明确点名：`LX04`、`X10A`、`L05B`、`L05C` 可能需要开启，否则可能出现“终端有回答，但音箱不播报”。
- 这个失败表现和源码一致：`text_to_speech()` 的异常被直接吞掉了，所以你很可能只看到终端输出，没有语音。
- Docker 部署是否更容易遇到这个问题，源码里没有任何专门分支，代码无法证明。

### 6. `mute_xiaoai` 的副作用

- 代码里的真实动作很直接：在满足提问条件时，如果检测到设备正在播放，就调用 `player_pause()` 强行暂停。
- 这意味着它不只是“打断小爱原始回答”，也可能打断当时设备上正在播放的别的内容。
- 源码里没有看到任何“某些型号上更不稳定”的特殊处理，所以稳定性和体验差异代码无法证明。
- 默认值有一处不一致：`Config` 里的默认值是 `false`，但 `xiao_config.yaml.example` 和 README 表格写的是 `true`。
- 如果你走 `config.yaml` 示例，默认更像“建议开”；如果你完全不传配置文件，代码默认其实是关的。

### 7. 持续对话相关字段的真实行为

- `start_conversation` 和 `end_conversation` 只是在内存里切换一个 `in_conversation` 布尔值。
- 源码里没有超时机制；只要进程不退出、你也没说结束持续对话，它就一直保持该状态。
- 上下文保留范围不是统一规则，而是取决于具体 bot。
- 对大部分 OpenAI 兼容 bot，历史记录走 `ChatHistoryMixin`，会保留“第一条提示词”加上最近 5 轮问答。
- `GeminiBot` 走的是 SDK 自己的 `start_chat()` 会话，并在历史长度大于 10 时裁剪。
- `LangChainBot` 走 `ConversationBufferWindowMemory`，但仓库里没有显式设置窗口大小。
- Docker 或进程重启后，会话状态一定丢失，因为这些状态都只存在内存里，没有任何持久化。

### 8. 当前是否原生支持 Gemini 的 Google Search

- 结论：当前仓库没有接入 Gemini 原生 `google_search`。
- `GeminiBot` 只创建了 `GenerativeModel`，然后直接 `send_message()`，没有 tools、grounding 或 search 配置。
- 当前仓库里唯一明确的检索实现是 `LangChainBot`，它走的是 `SerpAPIWrapper`。
- 所以如果你想用 Gemini 原生搜索，当前版本必须改代码。
- 如果不改代码，现成可用的“搜索”路径就是 `LangChain + SerpAPI`，不是 Gemini bot。

### 9. 如果使用 Gemini，实际支持到什么程度

- 当前 bot 类型里确实支持 `gemini`。
- 模型名支持到什么程度，代码层面是“字符串透传”：`gemini_model` 为空时默认用 `gemini-2.0-flash-lite`，否则把你填的字符串直接交给 SDK。
- `gpt_options` 对 Gemini 基本无效。虽然 `MiGPT.ask_gpt()` 会把 `gpt_options` 传进去，但 `GeminiBot.ask()` / `ask_stream()` 根本没有使用这些参数。
- `gemini_api_domain` 的用途很明确：给 Gemini SDK 指向一个自定义 REST 域名/代理，不是普通的模型参数。
- Docker 环境里没有任何 Gemini 专用网络设置逻辑。
- 另外要注意：通用 `proxy` 配置在 GeminiBot 里也没有被使用，所以如果 Gemini 访问需要代理，当前仓库没有给 Gemini 单独做适配。

### 10. 搜索结果在小爱播报场景下如何呈现

- 代码没有“搜索结果播报模式”这种专门逻辑。
- 搜索结果最终仍然只是普通文本，走统一的 `_normalize()` 和 TTS 流程。
- `_normalize()` 会把换行替换成逗号、把引号替换成逗号、去掉 `*`，空格还会被替换成 `--`，并不适合朗读长来源列表。
- 代码也没有“保留引用来源”或“只播报结论”的开关。
- 如果你要把搜索结果拿去播报，结论应该靠 prompt 约束；源码本身不会帮你压缩到适合语音的格式。

### 11. 官方镜像和版本固定策略

- 仓库 README 指向的官方镜像是 `yihong0618/xiaogpt`，并写明是 X86/ARM 共用镜像。
- 但 README 没有给出推荐固定 tag，也没有告诉你应该 pin 到哪个版本。
- 仓库里的 Docker 示例全部直接写镜像名，没有做版本固定。
- 所以“固定哪个 tag”这个问题，当前仓库代码和 README 都没有直接答案。
- 如果你追求可复现部署，这部分需要你在实际部署时自行选定并 pin 版本。

### 12. 配置文件挂载路径与目录结构

- 从程序本身看，它只关心你传给 `--config` 的文件路径，所以单文件挂载和整目录挂载都能成立。
- 仓库官方 Docker 示例走的是整目录挂载到 `/config`，然后传 `--config=/config/config.yaml`。
- Dockerfile 里也声明了 `VOLUME /config`，所以整目录挂载更贴近官方示例。
- 源码里没有专门的日志目录，也没有专门的缓存目录挂载要求。
- 但有一个容易忽略的点：小米登录态文件路径是 `~/.mi.token`，不是 `/config`。如果你需要它在容器重建后继续存在，应该额外挂载容器 home 目录里的这个文件。

### 13. 容器启动参数的最小集合

- 如果 `config.yaml` 已经写全，单从代码看，`--config=/config/config.yaml` 就可以作为主入口。
- 只有当你不想把密钥写进配置文件时，才需要依赖环境变量回填，例如 `OPENAI_API_KEY`、`GEMINI_KEY` 等。
- 如果你使用第三方 TTS，而不是 `tts: mi`，还需要额外考虑 `XIAOGPT_HOSTNAME` 和 `XIAOGPT_PORT`。
- 源码里没有任何时区、本地时间挂载相关逻辑，因此不是必需项。

### 14. 重启策略和健康检查

- 当前仓库没有 Compose 文件，也没有内置 `restart` 策略。
- Dockerfile 里没有 `HEALTHCHECK`。
- 程序内部有少量重试和重新初始化逻辑，但那只是针对轮询小爱接口时的局部错误，不是完整的进程级自愈。
- 如果容器异常退出，是否自动拉起完全取决于你自己的 Docker / Compose 配置，不是项目本身提供的能力。

### 15. `tts: mi` 是否已经足够

- 从代码路径看，`tts: mi` 是最简单、额外依赖最少的一条路。
- 它不需要本地 HTTP 语音服务，也不需要开放 `9527` 端口。
- 如果以后换第三方 TTS，当前实现会在本机或容器里启动一个 HTTP 服务，把音频 URL 喂给小爱播放。
- 这也是为什么 README 明确要求第三方 TTS 在 Docker 下映射 `9527`，并设置 `XIAOGPT_HOSTNAME`。
- 至于你当前设备上小米原生 TTS 是否稳定，代码无法证明。

### 16. 回答长度是否需要专门约束

- 代码没有任何硬性的长度裁剪，长度控制几乎完全靠 `prompt`。
- 当前源码默认 prompt 是“请用 300 字以内回答，请只回答文字不要带链接”；`xiao_config.yaml.example` 也是 300 字，不是 50 到 100 字。
- README 表格里写的 100 字以内，和当前代码并不一致。
- 由于语音输出前只做了非常粗糙的文本归一化，分点、长段落、引用、代码块都不会被优雅处理。
- 所以如果你的重点是语音播报体验，确实应该单独设计更短、更口语化的 prompt，但这是 prompt 设计问题，不是代码自动帮你完成的事情。

### 17. 日志和排障方式

- 当前项目没有文件日志，主要就是标准输出和标准错误。
- 在 Docker 里，这基本就等于“看容器日志”。
- `verbose` 主要控制日志级别；`-v` 会进到 DEBUG，`-vv` 还会把轮询细节打出来。
- 因此“容器日志”和“xiaogpt 自身 verbose 输出”并不是两套系统，本质上是同一条输出链路。
- 默认是否开启 `verbose`，从代码看更适合只在排障时临时打开，因为高等级日志会明显更吵。

### 18. 配置热更新能力

- 代码只在启动时读取一次配置文件，没有文件监听，也没有热更新逻辑。
- 所以修改 `config.yaml` 后，结论就是：必须重启进程或容器。
- 当前运行中能动态修改的，主要只有 prompt 和持续对话状态这类内存变量。
- 动态修改的 prompt 会覆盖当前进程里的 `self.config.prompt`，但重启后仍然会回到配置文件内容。
- 换句话说：运行时 prompt 优先级只在“当前进程、当前生命周期”里更高；一旦重启，静态配置重新生效。

### 19. 小米登录状态的长期稳定性

- “cookie 会不会过期”“账号密码方式会不会更容易失效”这类结论，源码无法证明。
- 但源码能证明两件事：一是项目明确依赖 `~/.mi.token` 这个登录态文件；二是每次启动都会走一次 `MiAccount.login("micoapi")`。
- 这说明长期运行时，登录态持久化不是可有可无的附属项，而是部署层面需要认真处理的内容。
- 如果容器重建后没有保留 `~/.mi.token`，而你又指望只靠 `cookie` 顶住，当前代码路径并不稳妥。

## 十一、基于源码，部署前最值得先确认的 5 件事

1. 先确认真实的 `hardware` 和 `mi_did`，否则启动阶段就可能过不去。
2. 再确认登录方案是不是同时覆盖了 `~/.mi.token` 的持久化；当前代码里 `cookie` 不是完整免登录。
3. 确认你的设备是否要开 `use_command`，以及是否只能走 `tts: mi`。
4. 如果准备用 Gemini，先接受一个前提：当前仓库没有 Gemini 原生 Google Search；要么不用，要么改代码。
5. 如果未来要换第三方 TTS，提前把 `9527` 端口映射和 `XIAOGPT_HOSTNAME` 一起纳入 Docker 方案。
